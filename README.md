# Personal LLM Knowledge Base (NotebookLM Hybrid)

A small, opinionated system for turning NotebookLM Markdown exports into a tightly
linked personal wiki, maintained by an LLM (Claude Code) in an interactive session.
The system itself is shareable; the notes inside it are not.

---

## Two-repo privacy model (read this first)

This project is **two independent git repositories**, on purpose:

| Repo | Root | Tracks | Remote |
|---|---|---|---|
| **System** | `knowledge-base/` | `CLAUDE.md`, `README.md`, `scripts/`, `templates/`, `launchd/`, partial `.obsidian/` config, `.gitignore` | GitHub (safe to publish) |
| **Content** | `knowledge-base/content/` | `raw/` (NotebookLM exports) and `wiki/` (generated pages) | **Local only**, or a *private* remote / backup target |

Why two repos instead of one repo plus `.gitignore`:

1. The system repo's `.gitignore` excludes `content/`, so `git add -A` at the root will
   never stage private notes.
2. Because `content/` is itself a git repo, git treats it as a submodule-style boundary
   and refuses to recurse into it. Even if the ignore rule were removed by accident,
   the public repo still cannot accidentally absorb the wiki.

**Hard rule:** `content/` must never be added, committed, or pushed under the system
repo. Do not add a remote to the content repo unless that remote is private (a private
GitHub repo, an external drive, or Time Machine).

## Folder layout

```
knowledge-base/                         # SYSTEM repo  → GitHub
├── .gitignore
├── CLAUDE.md                           # operating instructions for Claude Code
├── README.md                           # this file
├── scripts/
│   ├── maintain.sh                     # end-of-day maintenance helper
│   └── maintenance-prompt.md           # generated by maintain.sh (gitignored)
├── templates/
│   ├── source.md                       # one page per ingested NotebookLM export
│   ├── concept.md                      # one page per concept extracted
│   └── entity.md                       # one page per named entity (person/org/tool/…)
├── launchd/
│   └── com.user.kb-maintenance.plist   # optional 18:00 reminder, NOT autonomous editing
├── .obsidian/                          # vault root = knowledge-base/
└── content/                            # CONTENT repo → LOCAL ONLY
    ├── .gitignore
    ├── raw/                            # NotebookLM Markdown exports (immutable source)
    └── wiki/
        ├── index.md                    # the map — always read first
        ├── log.md                      # ingestion + maintenance history
        ├── concepts/
        ├── entities/
        └── sources/
```

## Ingestion workflow

1. Drop a NotebookLM Markdown export into `content/raw/`.
2. Open an interactive Claude Code session at the system root (`knowledge-base/`).
3. Ask it to ingest new files. Per `CLAUDE.md`, Claude will:
   - Read each new file in `content/raw/` thoroughly.
   - Create a summary page in `content/wiki/sources/` from `templates/source.md`.
   - Create or update concept pages in `content/wiki/concepts/` and entity pages in
     `content/wiki/entities/` from their respective templates.
   - Add bidirectional `[[wikilinks]]` between the new source page, its concepts, and
     its entities.
   - Update `content/wiki/index.md`.
   - Append `## [YYYY-MM-DD] ingest | <filename>.md` to `content/wiki/log.md`.
4. Review with `git -C content diff` and commit with
   `git -C content commit -am "ingest: <filename>"` if satisfied, or
   `git -C content restore .` to discard.

**Anything inside `content/raw/` is treated as immutable source.** Claude is instructed
never to modify it.

## Manual end-of-day maintenance routine

Maintenance is owner-triggered. There is **no headless `claude -p` job and no
launchd-triggered editing**. The flow:

1. From the system root, run `./scripts/maintain.sh`.
2. The script generates `scripts/maintenance-prompt.md` (with today's date substituted)
   and copies it to the clipboard.
3. Open an interactive Claude Code session at the system root and paste the prompt.
4. Claude reads `content/wiki/index.md`, performs a full lint pass (contradictions,
   stale info, orphans, broken `[[wikilinks]]`, missing connections, merge
   opportunities), makes surgical changes, appends a maintenance entry to `log.md`,
   and summarises the diff.
5. Review with `git -C content diff`. Either:
   - `git -C content commit -am "maintenance: YYYY-MM-DD"` to keep the changes, or
   - `git -C content restore .` to discard them.

Always commit the content repo *before* a big ingestion or maintenance pass so there
is a clean rollback point.

## Setup (one-time)

```bash
# Scaffold and create files (this is what the build does).
chmod +x scripts/maintain.sh

# Initialise the content repo (PRIVATE, local-only by default).
cd content
git init
git add -A
git commit -m "init: content vault (raw + wiki)"
cd ..

# Initialise the system repo (publishable).
git init
git add -A
git commit -m "init: knowledge base system core"

# Privacy verification — must print nothing.
git ls-files | grep '^content/' && echo "!! ABORT: content tracked in system repo" || echo "OK: content not tracked"
```

Then, when ready to publish the system repo:

```bash
# Create an EMPTY repo on GitHub (e.g. "knowledge-base-system"), then:
git remote add origin <your-repo-ssh-url>
git push -u origin main
```

Do **not** add a public remote to the content repo.

## Optional: launchd reminder

`launchd/com.user.kb-maintenance.plist` is a notification-only LaunchAgent that posts
a "Run end-of-day KB maintenance" notification at 18:00 each day. It does not run
Claude or touch the wiki. To install:

```bash
cp launchd/com.user.kb-maintenance.plist ~/Library/LaunchAgents/
launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.user.kb-maintenance.plist
```

If the Mac is asleep at 18:00 the notification fires on wake; if powered off, that
day is skipped.

## What's intentionally not here

- No headless `claude -p` automation — maintenance is manual by design.
- No second remote on the content repo by default — opt in only.
- No GitHub Actions, no CI, no auto-push — this is a personal tool.
